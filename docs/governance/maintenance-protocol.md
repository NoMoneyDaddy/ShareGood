# 制度維護協議

> 讀者：未來所有 session 的主模型。目的：讓制度檔能安全地演化——可以更新，但不能被弱模型
> 一時的判斷悄悄改壞。

## 1. 分級授權：什麼可以自己改，什麼要先問

### 綠區（自行修改，commit 訊息說明即可）
- `docs/governance/lessons/` 新增教訓檔、更新既有教訓。
- `master-plan.md`：勾選進度、為 M5–M8 補細部規格（格式比照 M0–M4）、修正錯字與失效路徑。
- `delegation-templates.md`：微調措辭、補「常見失敗提醒」。
- `CLAUDE.md` 的「目前階段」清單勾選。

### 黃區（可以提議並準備好 diff，但**執行前必須經使用者同意**）
- `CLAUDE.md` 硬規則的增刪改。
- `model-dispatch.md` 的門檻數字與升降級邏輯。
- `judgment-rubrics.md` 的判準本身（新增範例屬綠區）。
- `master-plan.md` 的已定決策：non-goals、技術棧、milestone 砍線、上線 gate。
- 刪除任何制度檔或整節內容。

### 紅區（任何模型都不做）
- 修改 `docs/plan/original-master-plan-v1.md`（歷史備份，唯讀）。
- 在未經使用者同意下，把制度檔內容改寫成與使用者已確認決策相反的方向。

## 2. 改檔前的備份規則

- 綠區小改（新增、勾選、錯字）：git 歷史即備份，直接改。
- 黃區或任何**重寫超過一節**的修改：先 `cp 檔案 檔案.bak-YYYYMMDD` 放同目錄，改完、使用者確認後
  刪 `.bak`。目的：session 中途壞掉時，工作目錄裡永遠有上一版。

## 3. 教訓（lessons）怎麼寫

**時機**：花超過 30 分鐘才解掉的問題、被使用者糾正的做法、重複踩到第二次的坑——解決後立即寫，
不要「等 session 結束再整理」（session 結束往往就是忘記的時候）。

**格式**（一課一檔，放 `docs/governance/lessons/`）：
```markdown
# 一行結論（讀這行就該知道要怎麼做）
- 日期：YYYY-MM-DD
- 情境：什麼任務、什麼環境下遇到
- 症狀：看到什麼錯誤／什麼行為
- 原因：真正的原因（不是猜測；還沒確認就寫「未確認，目前假設是…」）
- 修法：具體怎麼解的（指令、設定、程式碼位置）
- 引申規則：下次遇到類似情況的判準（可省略）
```
**檔名**：`YYYYMMDD-短橫線分隔的主題.md`，例：`20260810-zeabur-minio-presigned-url.md`。

**不要寫進 lessons 的**：repo 或對話紀錄本來就有的資訊、一次性的手滑、對制度的修改建議
（那走 §1 黃區流程）。

## 4. 精簡與合併（防制度膨脹）

- `lessons/` 超過 **20 檔**：開一個整理任務——把過時的刪除、同主題的合併、
  升格為通則的（重複驗證過的模式）走黃區流程提議寫進對應制度檔，然後刪掉原 lesson。
- `CLAUDE.md` 超過 **80 行**：必須瘦身。方法只有一種：把細節下放到 `docs/governance/` 對應檔，
  CLAUDE.md 留路由。不允許「因為重要所以留在 CLAUDE.md」——路由表就是重要性的入口。
- 制度檔之間出現**互相矛盾**：以 CLAUDE.md 硬規則 > model-dispatch > 其他 的優先序臨時裁決，
  同時立刻回報使用者裁決並修檔（矛盾本身寫一課 lesson）。

## 5. 每次 session 結束前的固定動作（checklist）

- [ ] 未落檔的結論與決策 → 落檔。
- [ ] 該寫的 lesson 寫了。
- [ ] `CLAUDE.md`「目前階段」與 `master-plan.md` 進度勾選更新了。
- [ ] 全部 commit 並 push。
- [ ] 有未完成事項 → 用一段話寫清楚「下個 session 開場第一句該說什麼」，
      放進交接（回覆使用者，或視情況更新 `letter-to-future-sessions.md` 的交接區）。
