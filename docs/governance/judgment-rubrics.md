# 判斷力 Rubrics：拿不準時翻這份

> 讀者：主對話模型。用途：把「需要品味的判斷」變成可查表的判準。
> 每條判準附一個正例（✅ 這樣判是對的）和一個反例（❌ 這樣判是錯的），範例都用 ShareGood 場景。
> 查表後仍拿不準 → 這件事本身就是「該升級模型或該問使用者」的訊號。

---

## 1. 何時該升級模型

**判準：任務的「判斷密度」高於「執行密度」，或錯誤成本高且事後難驗證，就升級。**

升級訊號（命中任一條就升）：
- 同一子任務在目前等級已失敗達上限（Haiku 1 次、Sonnet 2 次，見 model-dispatch.md §6）。
- 任務要求在多個合理方案間取捨，而取捨依據不在任何規格檔裡。
- 產出會被大量後續工作依賴（資料模型、API 慣例、制度檔），改起來牽一髮動全身。
- 你發現自己在猜使用者意圖，而且猜錯的代價超過一個 session。

✅ 正例：要決定 `items` 狀態機是否合併 `reserved` 與 `handover_pending` 兩個狀態。這影響所有後續
　查詢與 UI，規格書沒寫死取捨理由 → 派 opus 出第二意見，或直接問使用者。
❌ 反例：Sonnet 寫一個分頁查詢寫錯了 offset 計算，就說「這任務需要 Opus」。這是執行錯誤不是判斷
　缺口，帶著錯誤訊息讓 Sonnet 再修一次即可（第 2 次再錯才升級）。

## 2. 何時算真的完成

**判準：「完成」＝驗收條件逐條有證據，且驗收者不是實作者。缺任何一項就不是完成，是「待驗收」。**

完成的必要條件：
- 驗收清單逐條打勾，每勾附證據（測試輸出／實跑結果／read-back 確認）。
- 程式碼類：功能被實際執行過（不是「編譯通過」「型別檢查通過」）。
- 沒有留下未說明的 TODO、被註解掉的失敗測試、或「之後再補」的空函式。
- 對使用者的回報中，做了什麼與沒做什麼分開列，沒做的明說。

✅ 正例：「留言需要 API 完成：`POST /api/items/[id]/claims` 已實作；fresh agent 實跑 curl 建立留言
　成功（回 201）、重複留言被 unique constraint 擋下（回 409）、未登入回 401；測試 3 條全綠（輸出附後）。」
❌ 反例：「留言功能已完成，程式碼在 `app/api/claims/route.ts`，邏輯上涵蓋了所有情況。」——沒有任何
　實跑證據，「邏輯上」是未驗證的訊號。這要當成未完成退回。

## 3. 何時該停下來問使用者

**判準：問題滿足「只有使用者答得了」＋「猜錯代價高於等待」兩個條件才問；否則自己決定並記錄。**

必須問的情境：
- 不可逆或對外的動作：刪資料、force push、對外發布、動到正式環境、花錢（升級方案、買服務）。
- 需求本身有兩種以上合理解讀，且做錯要重工超過半天。
- 要違反或修改既有制度檔／master-plan 的已定決策時。
- 涉及使用者個人偏好（品牌文案語氣、UI 風格取捨）且無既有慣例可循。

不該問的情境（自己決定，並在回報中記錄決定與理由）：
- 技術實作細節在慣例檔已有答案，或業界有明確主流做法。
- 可逆、低成本的選擇（先選一個，錯了改回來比等回覆便宜）。

✅ 正例：master-plan 說「原圖預設不保留」，但使用者臨時要求做「圖片編輯」功能，兩者衝突 → 停下來問，
　因為這是規格層級的矛盾，只有使用者能裁決。
❌ 反例：「請問分頁每頁要 20 筆還是 25 筆？」——可逆、低成本、有慣例（master-plan 已定 20），
　問這個是把決策成本丟回給使用者。自己照慣例做。

**問的方式**：一次把同一批問題收齊再問（使用者規則：開場最多一批），每題附建議選項與你的推薦，
不要開放式地問「你覺得呢」。

## 4. 什麼訊號代表方向錯了，該換路而非重試

**判準：失敗的「原因類型」不變但你只是更用力重試 → 停。換路是換方法，不是換措辭。**

換路訊號（命中兩條以上，立刻停手換路）：
- 同一個錯誤訊息出現第 3 次，而你前兩次的修法本質相同。
- 每修一處就冒出新的錯（打地鼠模式），錯誤數量不收斂。
- 你開始寫「繞過」而不是「修好」：hack、猜測性的 workaround、註解掉失敗的檢查。
- 動的檔案數量遠超當初預估（預估 2 檔，已動 8 檔還沒完）。
- 你已經無法一句話說清楚「現在這步是為了什麼」。

換路的動作：
1. 停手，把目前狀態寫下來（試了什麼、卡在哪、目前猜測）。
2. 考慮 revert 到上一個乾淨 commit（隨做隨 commit 的價值就在這裡）。
3. 換一條路：不同的技術方案、更小的任務切法、或升級模型帶著失敗軌跡重新設計。

✅ 正例：為了讓 MinIO presigned URL 在 Zeabur 內外都能用，連改三次 endpoint 設定都失敗 →
　停下，改用「上傳走 server-side proxy」的替代方案重新設計，而不是第四次調整 DNS 假設。
❌ 反例：測試一直紅，於是把該測試改成 `skip` 讓 CI 變綠。這不是換路，是把問題藏起來，
　嚴格禁止。

## 5. 品質底線怎麼驗（ShareGood 專用 checklist）

**判準：以下三組底線，任何 milestone 驗收時逐條檢查；不過就不算完成，與功能是否能動無關。**

### 5a. 安全底線
- [ ] 每個 mutation API 有 server-side 權限檢查（不能只靠前端隱藏按鈕）。驗法：以另一個使用者
      身分 curl 打別人的資源，必須拿到 403/404。
- [ ] 沒有任何 secret 寫死在程式碼裡（grep `sk-`、`password`、`secret` 掃過）。
- [ ] 使用者輸入有 escape/sanitize（留言、物品描述試貼 `<script>` 不得執行）。
- [ ] 上傳檔案驗 magic bytes 與大小上限，不是只看副檔名。

### 5b. 資料與容量底線
- [ ] 圖片與大檔 binary 不進 PostgreSQL（DB 只存 object key）。
- [ ] 所有列表查詢有分頁（含後台），禁止 `SELECT *`。
- [ ] 有寫入的地方就有對應索引或 unique constraint（照 master-plan 索引清單）。
- [ ] 新增的每張表都在 master-plan 資料表清單裡（不得擅自加表；要加先更新規格）。

### 5c. 可維運底線
- [ ] 管理員的每個操作寫 audit log。
- [ ] 失敗路徑有處理：上傳失敗、DB constraint 衝突、第三方（OAuth）掛掉時使用者看到明確錯誤而非白屏。
- [ ] migration 可以在乾淨 DB 上從零跑通（驗法：fresh agent 起乾淨環境跑 `prisma migrate` + seed）。

✅ 正例：M1 驗收時功能全部能動，但發現 `PUT /api/items/[id]` 沒檢查 owner → 判定不通過，
　先修權限再談完成。
❌ 反例：「這只是 MVP，安全之後再補」——安全底線不隨 MVP 縮水，這正是弱模型最常見的合理化話術。

## 6. Harness 的極限（誠實條款）

以上 rubrics 補得了**執行品質**（拆解、驗證、多樣本評審）。以下兩類**補不了**，遇到就承認：

1. **模糊題／品味判斷**：品牌語氣、UX 微妙取捨、「這個功能使用者會不會喜歡」。
   → 出路按序：(a) 升級到 Opus 出多方案＋各方案的 trade-off，(b) 把選擇權連同 trade-off 交給
   使用者，(c) 明說「這題模型判斷不可靠，建議找真人使用者試」。禁止假裝有把握。
2. **需要外部事實而查不到**：法規、費率、第三方服務的最新限制。
   → 查得到就查（派 research agent），查不到就標註「未確認」並說明依據什麼假設，不要編造。
