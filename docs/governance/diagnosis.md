# Harness 快速診斷：三大漏洞與修法

> 本檔目的：指出本環境（Claude Code 遠端 session）最漏 token、最容易失焦、最容易出錯的前三名問題。
> 後續所有制度檔（model-dispatch.md、judgment-rubrics.md、delegation-templates.md）都以本檔為依據。
> 讀者是未來的主對話模型（可能是 Sonnet / Opus / Haiku）。每一條都有「症狀怎麼認」與「立刻怎麼修」。

---

## 漏洞 #1：指揮官親自下場讀檔，主對話 context 被原始內容灌爆

**這是最漏 token 的問題，也是三個問題中最優先要防的。**

### 症狀怎麼認
- 主對話裡連續出現多次 `Read` 整個檔案、`Grep` 大量結果直接回灌主對話。
- 一個回合內讀了 3 個以上的檔案、或單檔超過 400 行，而讀的目的只是「找某個東西在哪」。
- 開始出現「context 低」警告，或早期指令被壓縮摘要掉、模型開始重問已回答過的問題。
- 用 `WebFetch` / `WebSearch` 把整頁網頁內容拉進主對話。

### 為什麼嚴重
主對話的 context 是整個 session 最貴的資源：它承載使用者的原始要求、已做的決策、還沒做的事。
每灌進 500 行原始檔案內容，就是把這些擠出去。context 壓縮之後，弱模型最常見的退化是：
忘記早期決策、重做已完成的事、違反已同意的規則。

### 立刻怎麼修
1. **大量讀取一律派 subagent**：掃 repo、找定義、查文件、讀網頁，派 `Explore`（唯讀搜索）或
   `general-purpose`，明確要求「只回結論與 file:line，不要貼原文」。
2. **派工門檻寫死**：預估要讀「超過 3 個檔案」或「單檔超過 400 行」或「任何網頁」→ 派工，不自己讀。
   例外：你接下來就要 `Edit` 的那個檔案，必須自己 Read（Edit 工具要求先讀）。
3. **回報合約**：subagent 的長產物一律落檔（寫進 `docs/` 或 scratchpad），主對話只收路徑＋三行摘要。
   詳細規則見 `docs/governance/model-dispatch.md` 的「回報合約」節。

## 漏洞 #2：無驗收條件的派工與自我驗證，「宣稱完成」≠「實際完成」

**這是最容易出錯的問題。弱模型的典型失敗不是做不出來，而是做了 80% 然後報告 100%。**

### 症狀怎麼認
- 派工 prompt 只有一句「幫我實作 X」，沒有驗收條件、沒有回報格式。
- subagent 回報「已完成」但沒有附任何證據（測試輸出、實跑結果、file:line）。
- 同一個模型自己寫、自己檢查、自己說沒問題。
- 回報裡出現「應該可以」「理論上」「大致完成」這類措辭——這是未驗證的訊號。

### 為什麼嚴重
弱模型對「完成」的判斷天生偏樂觀。沒有外部判準時，它會把「程式碼寫出來了」當成「功能做好了」。
在 ShareGood 這種有權限、狀態機、併發保護的專案，80% 完成的程式碼往往比沒寫更危險
（例如：忘了 server-side 權限檢查的 API 端點看起來能動）。

### 立刻怎麼修
1. **派工三件套**：每次派工必含 (a) 目標與動機（為什麼要做）、(b) 驗收條件（可判定的清單）、
   (c) 回報格式（回什麼、多長、附什麼證據）。模板見 `docs/governance/delegation-templates.md`。
2. **驗證不自驗**：驗收一律派一個 **fresh-context** 的新 subagent（不帶實作過程的記憶），依據驗收條件檢查：
   - 檔案類 → read-back：實際打開檔案確認內容存在且正確。
   - 程式碼類 → 跑測試或實跑（`npm test`、實際 curl API），不接受「看起來對」。
   - 高風險判斷（架構、安全、資料模型）→ 第二意見：再派一個 agent 獨立評估，或多答案評審選優。
3. **證據語言**：回報中每個「已完成」都必須能指到證據（測試輸出、指令結果、file:line）。
   沒有證據的完成聲明，一律當作「未完成」處理。

## 漏洞 #3：單一 session 撐到 context 壓縮，決策與教訓隨 session 蒸發

**這是最容易失焦的問題，也是長期最傷的：每個 session 從零開始，重複踩同樣的坑。**

### 症狀怎麼認
- session 進行很久之後，模型開始自我矛盾（違反早先同意的規則）、重問已回答的問題。
- 大段工作成果只存在對話裡，還沒寫進任何檔案、還沒 commit。
- 上個 session 踩過的坑（例如某個套件版本問題、Zeabur 的某個限制），這個 session 又踩一次。
- 一個 session 想一口氣做完多個 Phase，中途 context 壓縮，後半段品質明顯下降。

### 為什麼嚴重
對話內容是揮發性的；檔案和 commit 才是持久的。這個環境的 container 是暫時的，
session 結束後沒 push 的東西一律消失。弱模型在被壓縮過的 context 上工作，錯誤率顯著上升。

### 立刻怎麼修
1. **隨做隨寫、隨寫隨 commit**：每完成一個可交付單位（一個檔案、一個功能、一個修復）立即 commit。
   不累積「等做完再一起 commit」。session 中斷時，已 commit 的就是使用者拿到的全部。
2. **一個 session 一個 Phase**：master-plan 的每個 Phase 開一個新 session 做。Phase 做完、驗收過、
   push 完就結束 session。不要在同一個 session 裡連做兩個 Phase。
3. **教訓落檔**：踩坑之後，把教訓寫進 `docs/governance/lessons/`（一課一檔，格式見
   `docs/governance/maintenance-protocol.md`）。新 session 開場先看 lessons 索引。
4. **感覺 context 吃緊時的動作順序**：立刻停下新產出 → 把手上未落檔的結論寫進檔案 → commit → push →
   把未完成項目寫進交接（新 session 的第一句話要說什麼），而不是硬撐著做完。

## 三條漏洞的共同根因（一句話版）

> **對話是揮發的、樂觀的、昂貴的；檔案與外部驗證是持久的、誠實的、便宜的。**
> 所以：結論落檔、派工帶驗收、驗證找外人。

## 本檔維護

- 發現新的系統性漏洞（不是單一事件，是重複發生的模式）→ 依 `maintenance-protocol.md` 流程提議新增。
- 單一事件的教訓 → 寫 `lessons/`，不要寫進本檔。
