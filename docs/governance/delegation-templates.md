# 派工 Prompt 模板

> 用法：派 subagent 時，挑對應模板**整段複製**，把 `【…】` 填空補齊後作為 Agent tool 的 prompt。
> 每份模板已內建「派工三件套」（目標與動機／驗收條件／回報格式）與回報合約。
> model 與投入度建議值來自 `model-dispatch.md` §4。注意：**Agent tool 沒有 effort 參數**，
> 投入度是寫在 prompt 文字裡的要求（「快速掃過」／「標準」／「務必徹底」）。拿不準就用模板標頭的預設。

---

## 模板 1：搜尋／盤點（subagent_type: `Explore`，model: haiku 或 sonnet，投入度：快速）

```
【目標與動機】
我在做【哪個 milestone / 任務】，需要先弄清楚【要找什麼】，因為【找到之後要拿來做什麼】。

【搜尋範圍】
- 目錄／檔案類型：【例：app/api/**、prisma/schema.prisma】
- 要找：【例：所有沒有呼叫 requireUser() 的 mutation route】
- 搜索廣度：【medium｜very thorough】

【驗收條件】
- 涵蓋整個指定範圍，不是找到第一個就停。
- 每個結果附 file_path:line。
- 明確說「找完了，共 N 處」或「以下範圍沒掃到：…」。

【回報格式】
只回結論，不貼大段原文。格式：
1. 一句話總結（共幾處、分幾類）
2. 清單：file_path:line — 一行說明
3. 沒把握的地方單獨列出
超過 30 條結果時：全文寫入【落檔路徑，例 /tmp/.../scan-result.md】，回報只給路徑＋前 5 條。
```
**常見失敗提醒**：Explore 找到部分結果就宣稱找完。驗收時抽查一兩個它沒列的位置。

## 模板 2：實作（subagent_type: `general-purpose`，model: sonnet［規格明確］/ opus［需補設計］，投入度：標準–徹底）

```
【目標與動機】
實作【功能名】。這是 master-plan.md §【節號】的一部分，目的是讓使用者能【使用者價值】。

【規格】
- 要做：【明確列出，或直接引 master-plan 對應段落全文貼給它】
- 不要做（scope guard）：【明確列出，例：不做通知、不動既有的 X】
- 必守慣例：master-plan.md §3（先讀）；所有 mutation 要 server-side 權限檢查；列表必分頁。

【驗收條件】（做完自己先逐條核對，但最終驗收由另一個 agent 執行）
- 【條件1，例：`npm test` 全綠，含新增的 N 條測試】
- 【條件2，例：實跑 curl POST /api/xxx 回 201；未登入回 401；打別人的資源回 403】
- 【條件3，例：prisma migrate 在乾淨 DB 跑通】
- type check 與 lint 通過（這是底線不是驗收）。

【回報格式】
1. 一句話：完成／部分完成（哪部分沒完成＋原因）
2. 改動檔案清單（file_path — 一行說明）
3. 驗收條件逐條：通過與否＋證據（測試輸出關鍵行／指令結果）
4. 你做的所有「規格沒寫、你自己決定」的決定，逐條列出
5. 上限 300 字（證據引文不計）
禁止：沒跑過就說「應該可以」；用「大致完成」掩蓋缺口。
```
**常見失敗提醒**：弱模型會「順手」做規格外的事（加抽象層、改無關檔案）。回報第 4 點就是用來抓這個。

## 模板 3：重構／批次修改（subagent_type: `general-purpose`，model: haiku［模式已定］/ sonnet，投入度：快速）

```
【目標與動機】
把【範圍】內的【舊模式】改成【新模式】，因為【理由】。模式已經確定，照做即可，不要自行變化。

【修改模式（含範例）】
改法：【描述】
範例——改前：
【貼一段真實的改前程式碼】
改後：
【貼對應的改後程式碼】

【範圍】
要改的位置清單：【file_path:line 清單，通常來自模板 1 的搜尋結果】

【驗收條件】
- 清單上每一處都改了；清單外一律不動（git diff 不得出現清單外檔案）。
- 改完 `npm test` 與 type check 全綠。
- 遇到跟範例長得不一樣、套不進模式的位置：跳過並回報，不要自己發明改法。

【回報格式】
1. 改了 N 處／跳過 M 處（跳過的逐條列出＋為什麼）
2. 測試與 type check 結果（貼關鍵輸出行）
```
**常見失敗提醒**：批次改最怕「自由發揮」。範例給得越具體，haiku 就越可靠；沒有範例不要派 haiku。

## 模板 4：研究／查文件（subagent_type: `general-purpose`，model: sonnet，投入度：標準）

```
【目標與動機】
我在做【任務】，需要弄清楚【問題】，答案會用來決定【什麼決策】。

【要查的問題】（要具體，不要開放式）
1. 【例：Zeabur Free Plan 目前的容量與 auto-sleep 限制是什麼？】
2. 【例：next-auth v5 在 App Router 的官方建議設定？】

【來源要求】
- 優先官方文件（可用 Context7 / WebFetch / claude-code-guide）。
- 每個結論標來源 URL 與查閱日期。
- 查不到就明說「查不到」＋你查過哪裡；禁止用訓練記憶編造版本號、價格、限制數字。

【回報格式】
每個問題：結論（1–3 句）＋來源＋信心（高／中／低）。
低信心的要說明為什麼低。全文超過 40 行就落檔，回報給路徑＋摘要。
```
**常見失敗提醒**：模型最愛在「版本號、價格、配額」上編造。凡是數字都要有來源，沒來源標「未確認」。

## 模板 5：審查／驗收（subagent_type: `general-purpose`，model: sonnet［常規驗收］/ opus［安全或架構審查］，投入度：徹底）

```
【目標與動機】
你是 fresh-context 驗收者，審查【什麼產出】。你沒有參與實作，這是刻意的——不要腦補實作者的意圖，
只看實際產出是否符合驗收條件。

【受審對象】
- 檔案／diff 範圍：【路徑或 git diff 指令】
- 對應規格：master-plan.md §【節】／【或貼驗收條件全文】

【驗收方法】（按產物類型執行，不可只用讀的）
- 文件 → read-back 逐條對驗收清單。
- 程式碼 → 實際執行：跑測試、起服務、curl 端點，把輸出貼進報告。
- 安全審查 → 依 judgment-rubrics.md §5a 逐條攻擊測試（用別人的身分打資源、貼 <script>、傳假檔案）。

【回報格式】
逐條驗收條件：PASS／FAIL／無法驗證（＋原因）。
每個 FAIL 附：問題描述、file_path:line、建議修法（一句話即可）。
最後一行結論：「通過」或「不通過（N 個 FAIL）」。
禁止：因為「整體看起來不錯」而放水個別 FAIL；把 style 偏好報成 FAIL（只驗清單上的東西）。
```
**常見失敗提醒**：驗收者兩種失敗——太鬆（沒實跑就 PASS）和太嚴（拿清單外的個人偏好刷 FAIL）。
模板最後兩行就是防這兩種。

---

## 通用規則（所有模板適用）

1. 填空時**寧可貼原文，不要轉述**：把 master-plan 對應段落直接貼進 prompt，比你摘要一次更不失真。
2. 一個 subagent 一個任務。兩件事就派兩個（獨立的並行派）。
3. subagent 回報不合回報格式（例如沒附證據）→ 退回要求補，不要自己腦補它做完了。
4. 長產物落檔路徑統一放 scratchpad 或 `docs/`，不要塞進對話。
