## 可執行規則清單

### 出處：sparanoid/chinese-copywriting-guidelines（中文文案排版指北）

1. 中文與半形英文字母之間須加一個半形空白。例：「在 LeanCloud 上，數據儲存是圍繞
   `AVObject` 進行的」；不可寫成「在LeanCloud上」。
2. 中文與阿拉伯數字之間須加一個半形空白。例：「今天出去買菜花了 5000 元」；不可寫成
   「花了5000元」。
3. 數字與其後的英文單位之間須加一個半形空白。例：「我家的光纖入屋寬頻有 10 Gbps」。
   **例外**：度數（90°）與百分比（15%）數字與符號之間不加空格。
4. 全形標點符號與其他字元（中文、英文、數字）之間不加空格。例：「剛剛買了一部
   iPhone，好開心！」；不可寫成「iPhone ，好開心」（逗號前面多一格）。
5. 不要重複使用標點符號加強語氣。例：用「德國隊竟然戰勝了巴西隊！」，不要寫成
   「戰勝了巴西隊！！！」。
6. 中文文案整體使用全形標點符號（，。「」？！、；：），不要用半形標點或英文彎引號
   （" "）取代中文引號。正確例：「嗨！你知道嘛？」；錯誤例：「嗨! 你知道嘛?」。
7. 純數字一律使用半形阿拉伯數字，不用全形數字。正確：「這件蛋糕只賣 1000 元」；
   錯誤：「賣 １０００ 元」。
8. 若整句話是引用的英文原文，該句內部維持半形標點（逗號、句號），不要因外層是中文
   語境就把英文句子內的標點改成全形。例：『Stay hungry, stay foolish.』，逗號句號
   維持半形，不寫成『Stay hungry，stay foolish。』。
9. 專有名詞／品牌名稱使用官方正確大小寫拼法，不要全部大寫或全部小寫。正確：
   「使用 GitHub 登入」；錯誤：「使用 github 登入」或「使用 GITHUB 登入」。此規則
   直接適用 ShareGood 站內的「Google」「LINE」「Telegram」「MinIO」等品牌名稱。
10. 不要使用不道地、非官方的英文技術詞縮寫。正確：「熟悉 TypeScript、HTML5」；
    錯誤：「熟悉 Ts、h5」。
11. 超連結文字與前後中文之間建議加空格，讓連結範圍在視覺上更清楚。例：「請
    [提交一個 issue](#) 並分配給相關同事」。此規則列為原文承認的「爭議用法」，
    非強制，但可作為連結密集頁面（如通知列表、說明頁）的排版參考。

### 出處：kevintsai1202/Humanizer-zh-TW（去 AI 腔判準）

這份資源不是排版規則，而是「一段文字讀起來像不像 AI 生成」的判準集合，適合用來
校對 ShareGood 面向使用者的說明文字、通知文案、系統提示語，避免出現生硬的機器腔調。

12. 避免高頻「AI 詞彙」堆疊，例如「此外」「至關重要」「深入探討」「突顯」「充滿
    活力」「寶貴的」。這類詞在中文口語中出現頻率遠低於 AI 生成文本，出現多個即為
    警訊，應改寫為更直白具體的說法。
13. 避免內容空泛地強調「意義」「影響深遠」「開啟新篇章」之類的誇大定性語句，
    尤其是功能說明或系統通知裡不需要這種修辭。
14. 避免用詞遮遮掩掩、迴避明確斷言（繫詞迴避），例如把「這是免費的」寫成「某種
    程度上可以視為一種無償形式的分享」。ShareGood 的說明文案應直接、明確。
15. 避免「不是...而是...」三段式對比句型過度重複使用；同一頁面內若連續出現多次
    這種句型，視為 AI 腔訊號，應改寫成不同句式。
16. 避免破折號與粗體的過度使用（例如整段話每句都用「——」或每個關鍵詞都加粗），
    這在中文網頁文案中會顯得不自然、像是機器排版。
17. 避免內文中插入表情符號（emoji）作為條列標題或語氣強調，尤其是正式的系統通知、
    法律聲明、表單錯誤訊息等情境。
18. 避免諂媚語氣或過度限定語（例如「當然，這只是一個小小的建議」「值得一提的是」
    這類純填充、不增加資訊量的開場白／收尾語），文案應直接進入重點。
19. 避免用「總之，這將為您帶來...」之類空泛正向的收尾句作結，若沒有具體內容支撐，
    應直接刪除或改寫成具體的下一步行動說明（例如「立即查看你的物品」）。

### 出處：Imbad0202/tw-formal-writing（僅取可遷移到一般網站文案的部分）

這份資源本質是「公文寫作規範」，多數規則（引敘語奉/准/據、稱謂語鈞/貴/本、期望語
請鑒核/請查照）屬於機關對機關的行文格式，與 ShareGood 面向一般民眾的鄰里共享
情境不符，**不採用**（詳見下節）。以下 2 條屬於可遷移的通用寫作原則與正字規範：

20. 核心寫作原則「簡、淺、明、確」——文案應簡潔、淺白、意思明確、用字確實，避免
    文言化或迂迴修辭。可直接作為 ShareGood 站內所有提示語、說明文字、錯誤訊息的
    校對判準：能刪掉的贅字就刪、能講白話就不要繞。
21. 法律統一用字（教育部標準正字），可直接套用做錯字校對：「布」非「佈」
    （公布、分布、頒布）；「帳」非「賬」（帳戶、帳目）；「黏」非「粘」（黏貼）；
    「計畫」作名詞時用「畫」、「規劃」作動詞時用「劃」；「紀錄」作名詞用「紀」、
    「記錄」作動詞用「記」；「徵」非「征」（徵收、徵稅）；「部分」「身分」用
    「分」不用「份」。

## 本站不採用的規則

- **Imbad0202/tw-formal-writing 的公文行文方向用語**（引敘語「奉／准／據」、稱謂語
  「鈞／貴／本」、期望語「請鑒核／請查照／請照辦」等）：這些是機關對機關、上行文／
  平行文／下行文的公文格式用語，套用到 ShareGood 這種面向一般民眾、強調溫暖鄰里感
  的共享平台文案上會顯得極度官腔、疏離，與 CLAUDE.md 明訂的「不官腔」品牌語氣直接
  衝突，故不採用。
- **Imbad0202/tw-formal-writing 的公文舊式贅詞清單本身雖不採用其「替代用語」，但其
  「應避免」的判斷可反向借用**：例如「為要」「為荷」「為禱」「實感德便」「至紉
  公誼」「鑒核示遵」「竊」「仰」「遵」「伏祈」這些詞本來就不會出現在網站文案裡，
  此處純粹是文言公文的歷史用語，不需要也不會被誤用，因此此清單本身對 ShareGood
  校稿沒有實際操作意義，僅第 20 條的「簡淺明確」原則被保留採用。
- **sparanoid/chinese-copywriting-guidelines 的「爭議用法」段落中『簡體中文使用
  直角引號「」』**：原文本身即標註這是簡體中文語境的用法建議，ShareGood 全站是繁體
  中文站台，引號使用遵循台灣繁中慣例（「」用於一般引用、『』用於引用中的引用），
  不採用簡體中文的直角引號規則另做調整。
- **TencentEdgeOne/awesome-website-prompts-and-skills 全部內容**：整個倉庫是網站
  生成用的程式碼／設計 Prompt 合集，沒有與文案撰寫規則相關的可用內容，故整份
  不採納，不勉強從中抽取規則。

## ShareGood 專用補充（依 CLAUDE.md 品牌語氣）

ShareGood 的品牌語氣要求是「溫暖、可信任、鄰里感、不賣萌、不官腔」。把這個定性要求
換算成校稿時可直接判斷的具體準則：

**避免的詞彙／句型**：
- 官腔／公文遺緒用詞：「爰」「茲」「業經」「揆諸」「渠」「乙份」「敬悉」「鑒核」
  「查照」等文言或公文專用字詞，在一般頁面文案、通知訊息、表單提示中一律不應
  出現（呼應上節「不採用公文行文用語」的判斷）。
- 行銷空話／AI 腔詞：「至關重要」「深入探討」「賦能」「打造」「開啟嶄新的一頁」
  「極致」「豐富多元」，這類詞在功能說明或系統通知裡通常沒有具體資訊量，應改寫
  成具體描述「做了什麼、使用者接下來要做什麼」。
- 賣萌／過度熱情語氣：疊字裝可愛（「棒棒」「讚讚」）、連續驚嘆號、大量表情符號
  堆疊在系統通知或表單錯誤訊息裡。ShareGood 是嚴肅處理鄰里互助與治理（檢舉、
  下架、限制功能）的平台，這類語氣會削弱可信任感，不應出現在交易/治理相關頁面；
  一般探索頁面可以有溫度但不誇張。
- 空洞道歉／推諉語：「很抱歉造成不便，我們會持續改進」這類無具體資訊的萬用道歉句，
  應改成明確說明原因與下一步（例如「這個物品已經被其他人領走了，看看其他好物吧」）。

**鼓勵的詞彙／句型**：
- 直接、口語化但保持禮貌的第二人稱敘述，像鄰居間對話，例如「你的物品」「分享給
  需要的人」「幫這位鄰居留言吧」，取代冷冰冰的「用戶」「該物品」「請進行操作」。
- 具體的下一步行動指引，尤其是通知、表單驗證錯誤、治理類頁面（檢舉／限制／申訴），
  要讓使用者清楚知道「發生了什麼」「現在能做什麼」，不用模糊帶過。
- 遇到治理與安全相關文案（下架原因、限制說明、申訴結果）時，語氣要溫和但清楚
  說明依據與後續，不必刻意軟化到含糊其詞，也不必生硬到像法院判決書；符合第 20 條
  「簡、淺、明、確」原則。

以上規則清單可直接交給後續校稿 agent 對照全站文案（頁面文字、按鈕、通知、表單
提示、Email/Telegram 通知內容等）逐條檢查。
