# 資料來源可行性盤點（S0–S5 分級）

> 查證方式：僅讀公開頁面與 robots.txt/sitemap，未登入、未呼叫任何 App API；遇 403 即停止並記錄，未嘗試繞過。查詢日期：2026-07-06。
>
> **共通現象：本次盤點的所有台灣官方優惠頁都沒有 JSON-LD 結構化資料、都沒有 RSS/Atom**——沒有任何一家提供機器可讀的優惠資料介面。

## 分級定義

- **S0**：官方 API／合作 API，可穩定整合
- **S1**：官方公開頁面，可人工收錄或低頻檢查，需保留來源與更新時間
- **S2**：第三方整理頁，僅供參考或人工確認，不建議自動同步
- **S3**：社群投稿來源，需人工審核與來源標註
- **S4**：會員 App／登入後內容，不可抓取，只能導流或使用者自行投稿
- **S5**：動態條碼、個人化券、一次性序號，禁止自動抓取或公開收錄

## 聚合功能三方案適配

**方案 A｜保守：只做 S3 使用者投稿**
- 全部來源的「內容」都能以使用者轉述形式進站（含 S4/S5 的 App 好康由使用者自述＋導流）。
- 投稿必附來源連結、人工審核、不貼券碼——沿用 M2 既有檢舉/黑名單/審核機制即可。
- 風險最低；內容冷啟動慢、時效差。

**方案 B｜中階：A ＋ S1 官方頁人工收錄與低頻檢查**（研究結論：建議 MVP 採用）
- 新增 10 個官方來源（#1,3,4,5,6,7,9,10,12,13）。
- 做法：編輯人工瀏覽→自寫摘要（不複製官方圖文）→附官方固定 URL 導流→顯示「來源＋查證日期」。
- 低頻異動偵測可半自動，但 #4 摩斯、#11 OPEN POINT 有反自動化措施、#3/#5/#12 是 JS SPA——這五個連異動偵測都建議純人工。
- 明確排除：所有 App 內個人化券、序號、動態條碼。

**方案 C｜進階：B ＋ S0 穩定整合**
- 立即可做：#17 政府超商門市資料（免費、月更、只需署名）＋ geocoding → 門市維度。
- 花錢可做：#18 Google Places API。
- **目前沒有任何一家速食/超商/量販在台灣提供公開的優惠資料 API**——S0 級「優惠內容」整合只能靠日後商務合作，短期不存在。

## 關鍵發現

1. 「麥當勞報報」已改版為「麥當勞APP」（2021），規格文件勿用舊名。
2. 家樂福台灣已 301 轉址至「萬家福 Uni-PROSPERITY」。
3. 麥當勞與肯德基 ToS **明文禁止重製轉載**——S1 收錄一律「自寫摘要＋導流」。
4. 友善食光/i珍食的**機制說明頁**是公開靜態內容（S1）、與惜食定位最契合；**即時庫存**都在 App 內（S4）只能教使用者自查。
5. 門市維度最務實路徑：政府開放資料（S0、免費）為主。
