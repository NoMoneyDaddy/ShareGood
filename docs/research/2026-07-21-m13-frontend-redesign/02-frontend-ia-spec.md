### 0.2 Pin 圖釘元件

```
.pin { width/height 依情境（22–36px）; border-radius: 50% 50% 50% 0; transform: rotate(-45deg);
       background: var(--pin-c, var(--brand)); box-shadow: 0 3px 6px rgba(shadow,.32); }
.pin-icon { transform: rotate(45deg); } /* 抵銷父層旋轉，圖示維持正向 */
.pin[data-fresh="new"]    /* 24 小時內：實色飽和 */
.pin[data-fresh="recent"] /* 1–3 天：實色偏淺（color-mix 62% + white）*/
.pin[data-fresh="old"]    /* 4 天以上：極淺底 + 內縮邊框，圖示變成描邊色而非填色 */
```

三段式新鮮度（`data-fresh`）用**明度深淺**表達，不是額外顏色，避免和「分類色」系統打架；色弱使用者
仍可靠圖示本身（每個分類固定圖示，不只靠顏色）辨識分類。`--pin-c` 由外層決定，即分類色 token
（`--cat-*`）；沒有分類色可用的情境（例如強制下架的「異常」圖釘）改用 `--danger` 並疊加虛線外圈
（`.pin-anomaly-ring`），與分類色系統明確視覺區隔（不會被誤讀成某個分類）。

### 0.3 Legend 圖例元件

`.legend-card` 是標準 `.panel`（`--surface` 底 + `--border` 框 + `--radius-m`）裡的內容排版慣例：
標題列（icon + 粗體小字）+ `.legend-grid`（響應式多欄，`repeat(auto-fill, minmax(150px,1fr))`）列出
「色塊（`.legend-swatch`，跟 `.pin` 同形狀但無陰影）+ 說明文字」。新鮮度圖例另用 `.legend-fresh-row`
橫向排列三顆縮小版 pin + 文字說明。**任何頁面只要引入了分類色或新鮘度深淺，就必須在同一畫面或
一鍵可達的地方放對應圖例**（不能只讓使用者自己猜色彩代表什麼）——這是延續定案預覽檔「為什麼不是
列表優先」卡片裡強調的「地圖做法必須自帶圖例才成立」的設計前提。

### 0.5 戳章（Stamp）元件

```
.stamp { 圓形; border: 2.5px solid var(--stamp-c); transform: rotate(-6deg); }
.stamp::after { inset:4px; border:1px solid var(--stamp-c); opacity:.55; } /* 內圈，仿印章雙線 */
.stamp.stamp-done    { --stamp-c: var(--brand);       border-style: solid;  background: var(--brand-tint); }
.stamp.stamp-active  { --stamp-c: var(--accent-deep);  border-style: dashed; background: var(--accent-tint); }
.stamp.stamp-pending { --stamp-c: var(--ink-faint);    border-style: dotted; opacity: .72; }
.stamp.stamp-void    { --stamp-c: var(--danger);       border-style: double; border-width:5px; background: var(--danger-tint); }
```

四種狀態**同時用顏色與邊框樣式（實心／虛線／點線／雙線）雙重編碼**，色弱使用者仍可靠邊框樣式分辨
——這是定案預覽檔明確強調的無障礙設計決策，任何新增的戳章狀態都必須遵守「顏色 + 邊框樣式」雙重
規則，不能只靠顏色。`.stamp-track` 是水平捲動軌道（手機版 `overflow-x:auto`），每個 `.stamp-step`
固定寬度、圖示 + 短標籤（等寬字體）+ 時間戳，`.stamp-connector` 依前一步是否完成切換實心/虛線。

**戳章語言的适用邊界（本文件的核心判斷之一，詳見各節「視覺套用」小節）**：戳章造型的敘事是
「蓋章＝留下不可逆紀錄、且狀態沿單一方向前進」，因此只適合**有限步驟、單向前進、且有審計軌跡
（audit log/history）佐證**的流程。凡是**可雙向循環**（例如 DealInfo 的 `published↔stale`）或
**沒有前台可視化流程頁面**的狀態機，都不套用戳章造型，改用既有的 `status-pill`／`Badge`
色塊語言（見 1.2 節）。逐頁判斷見各節。

### 0.6 手機版導覽：漢堡選單／抽屜（定案預覽檔的示範，本次前台範圍的落地方式）

定案預覽檔的 `.hamburger-btn` + `.nav-drawer`（`position:absolute; top:100%`，`max-height:
calc(100vh - 56px)` 可捲動，`768px` 以下顯示、`769px` 以上隱藏）示範的是「頂端導覽有『前台／後台』

`SiteHeader`（`src/components/site-header.tsx`）目前手機版右側只有主題切換鈕與通知鈴鐺，尚未

### 0.7 圖示庫決策

定案預覽檔用手刻 SVG `<symbol>` 定義圖示（`ic-furniture`／`ic-compass`／`ic-handshake` 等）。
ShareGood 現有程式碼庫全面使用 `lucide-react`（`02-current-state-design-audit.md` 已盤點確認），
**實作時不需要照搬定案預覽檔的手刻 SVG symbol 庫**，改用語意相近的 `lucide-react` 圖示填進
`.pin-icon`／`.stamp-icon`／`.legend-swatch` 等插槽即可（例如 `ic-furniture` → `Sofa` 或
`Armchair`、`ic-handshake` → `Handshake`、`ic-compass` → `Compass`，`lucide-react` 皆有對應圖示），
維持現有依賴管理慣例、不新增圖示庫。

  差自動判定 new/recent/old），取代目前純文字的分類標籤 pill（分類名稱文字仍保留在卡片下半部，

### 1.2 StatusPill（狀態徽章，輕量、清單/卡片層級用）

現有 `src/components/ui/badge.tsx`（shadcn Badge，variant: default/secondary/outline/destructive）
已被 `/me/wallet`、`/me/favorites`、`/support`、`/support/[id]` 廣泛使用。定案預覽檔的
`.status-pill`（色塊 + 前綴圓點 `::before`）視覈上更貼近地圖語言（圓點呼應圖釘造型），M13 建議：
**保留 shadcn Badge 元件本身（API 不變），只調整其色彩 token 與新增一個帶圓點前綴的視覺變體**，
讓 Badge 在需要「狀態感」的地方（物品狀態、回報狀態、優惠券狀態）都能加上前綴圓點，避免又造一個
平行的 pill 元件。圓點顏色沿用語意色（success=進行中偏正向、warning=待處理、danger=已下架/失效、
中性=已完成/終態），這一層色彩邏輯在各頁「狀態徽章」出現處保持一致。

**StatusPill 只用於清單、卡片、表格式的一列一物**；凡是「單一物件的多步驟進度」一律改用下面的
StampTrack，兩者分工明確、不重疊。

### 2.1 `/`（首頁）—— `src/app/(shell)/page.tsx`

**Top task**：讓訪客在 3 秒內看懂「這是什麼」，並且用最少點擊找到「跟我有關的好物」；對已登入
使用者則是「快速回到分享/接手動作」。**Primary action**：搜尋（訪客/一般瀏覽）與「我要分享」
CTA（已設定好 profile 的使用者）。

**視覺套用**：
- Hero 區（搜尋框 + 標題 + 分類捷徑 pill）**保留現有版面骨架**（M10 現況盤點已確認這塊設計完成度
  高，不需要重做），只換色系與圖示。分類捷徑 pill 目前是純文字圓角按鈕，**新增左側分類色小圓點**
  （不是完整圖釘造型，避免 hero 區過於花俏），讓使用者提早在 hero 就建立「分類=顏色」的心智模型，
  之後在 `/items` 與詳情頁看到同一顏色時能立刻對應。
- **新增「找好物地圖」分區入口區塊**（插在 hero 與「熱門好物」之間），這是定案預覽檔畫面 1
  最核心的落地：用 `db.city`（22 縣市）搭配 `db.item.groupBy` 依 `cityId` 算出各縣市目前
  `published` 物品數，渲染成 `.zone-grid`（ZoneTile，見 1.8 節），每個 tile 顯示縣市名 + 該縣市
  目前最多 3 個分類的代表圖釘（用該縣市物品數最多的前 3 個分類，各自的分類色）+ 物品總數。
  點擊 tile 直接導向 `/items?cityId=<id>`——**這是定案預覽檔反覆強調的「地圖做法：分區本身就是
  篩選，圖釘本身就是入口」，用來取代「先打開列表→選縣市→選分類→往下捲動」的傳統模式**，直接把
  `/items` 頁面本來就有的 `cityId` 查詢參數提前到首頁一鍵可達。物品數為 0 的縣市 tile 仍顯示但
  轉為低飽和「暫無物品」樣式（不隱藏，讓使用者知道平台涵蓋全台 22 縣市，只是這個縣市還沒人分享，
  這對於冷啟動期的縣市覆蓋率認知很重要）。
- 「熱門好物」網格套用 1.1 節 ItemCard 規格（新增分類圖釘）。
- 三步驟說明區塊、信任列、CTA 區**版面結構不變**（M10 現況盤點稱讚這裡避開了「三張卡片模板重複」
  的單調感，維持），只換色系為群青/金黃赭。

**降低操作摩擦**：新增的「找好物地圖」區塊本身就是摩擦削減——過去使用者要先進 `/items` 才能篩選
縣市，現在首頁一鍵直達已篩選好的列表，省掉一次「進頁面→操作篩選表單→送出」的完整往返。

**降低學習摩擦**：分類色點 + 縣市 tile 圖釘是漸進揭露的第一層——首頁只給「大致印象」（這個縣市
現在活躍嗎、有哪幾類東西），不要求使用者馬上理解完整圖例規則；完整圖例只在 `/items` 列表頁旁邊
才需要出現（見 2.2 節）。

**手機版轉換**：`.zone-grid` 在 480px 以下強制單欄（延續定案預覽檔 media query），維持現有
「拼貼圖只在 md 以上顯示」的既定決策（手機版 hero 只留搜尋 + 分類捷徑）。分區地圖區塊在手機版
改成可橫向捲動的單列 tile（而非直向堆疊全部 22 縣市造成首屏過長），只展示「目前登入者所在縣市」
＋「物品數前 5 名縣市」，附一個「看全部縣市」連結展開完整清單或導去 `/items`——這是本文件對定案
預覽檔（桌面版示範 6 個 tile 全部平鋪）在真實 22 縣市規模下的手機摩擦調整，避免手機使用者要
捲動 22 格才能到「熱門好物」區塊。

  （已經是很好的發現性設計，不需要更動）。

**降低學習摩擦**：

  （食品雜貨）分類選下去會展開標題為「即期食品規則」的子表單，即使分享的是離到期日很久的

- 法定警示（文創法 10-1／運動產業條例 24-1）與轉贈風險提示、即期食品食安提示**沿用 M10 記錄的

### 3.2 `/items/new/batch`（批量上架表單）—— `batch-item-form.tsx`

**Top task**：一次建立多筆相似物品，最小化「重複填同樣的縣市/分類」之類的重工。**Primary
action**：逐筆完成後一次送出。

**視覺套用**：不套用地圖/戳章語言，維持中性表單樣式。但**每一筆待上架項目的「就緒狀態」

實心 vs「未就緒」點線兩態）取代目前可能是純文字或圖示的就緒指示——這是判斷戳章語言**可以延伸
但只取其兩態子集**的例子：批量上架每一筆的完成度確實有「留痕」意義（使用者需要一眼掃過去知道

**降低操作摩擦**：說明文字已明講「優惠券/即期食品/電子票券/點數好康請個別到一般表單上架」，

範圍無關，維持現狀**。

**降低學習摩擦**：批量上架是進階功能，入口本身已經用「一次要分享好幾件物品？」白話問句取代
技術性名詞（不是直接寫「批量上架」讓使用者自己猜），**延續此文案策略**。

**手機版轉換**：多筆物品的表單在手機上必然需要分段——若目前是所有筆數平鋪在同一長頁，**建議
每筆物品做成可摺疊區塊**（預設全部展開第一筆、其餘摺疊摘要列 + 就緒狀態指示），避免手機使用者
在一個超長頁面裡迷失；點擊摘要列展開/收合單筆的完整欄位。這是本文件對批量表單手機體驗的

**共用元件**：BackBar（1.5，已用）。

### 3.3 `/items/[id]`（物品詳情頁）—— M13 最關鍵的一頁

**Top task**：因物品狀態而異——`published` 狀態的訪客是「決定要不要留言/申請」；`reserved`／
`handover_pending` 狀態的物主/接手者是「知道現在該做什麼、跟對方聯絡」；`completed` 狀態的任何人
是「回顧這次分享的完整記錄」。**Primary action**：對應上面三種情境分別是「留言/收藏/檢舉」、
「前往私訊/標記完成」、「留感謝/評分」。

**視覺套用（本文件的核心判斷）**：

M10 批次 2 已經把這頁重整成 4 個 Zone（分享者資訊列 → Zone 2 物品詳細資訊/quiet → Zone 3
互動與交接（單一卡片框，內含「現在可以做的事」提示）→ Zone 4 歷程/quiet），這個資訊架構
**本身是對的、M13 延續不變**，M13 要做的是把「交接進度」這個子區塊從純文字/既有時間軸，換成
定案預覽檔畫面 3 展示的 **StampTrack** 完整實作：

- 步驟固定為：`上架 → 留言認領（或直贈/抽籤，依成立方式二選一顯示）→ 物主確認 → 私訊敲面交時間
  → 完成交接`。每步驟的時間戳來自既有資料（`item.publishedAt`、`ClaimComment.createdAt`/
  `DirectShare.acceptedAt`、`Conversation` 首則訊息時間、`HandoverRecord.completedAt`），
  **不需要新增資料表**，純粹是既有欄位的視覺重組。
- 「留言認領」步驟依據物品實際成立管道換標籤文字（留言/直贈/抽籤三選一），這是既有邏輯
  （`lotteryActive`／`DirectShareSection`／`ClaimsSection` 互斥顯示）在戳章軌道上的鏡射呈現，
  不是新邏輯。
- 這個 StampTrack 只在 `handoverIsActive`（`reserved`/`handover_pending` 且是物主或接手者）與
  `showHistoryZone`（`completed` 狀態）兩種既有分支下渲染，**沿用現有的 `handoverIsActive`／
  `showHistoryZone` 布林判斷，不新增判斷邏輯**——`completed` 狀態下 StampTrack 全部節點顯示
  `stamp-done`（全部完成的收合式歷史記錄），視覺上是「蓋滿章的存根」，比純文字「已完成分享」
  更有「這件事真的走完全程」的儀式感，呼應貢獻值/感謝留言帶來的正向情緒收尾。
- **分類圖釘**：詳情頁頂端「分享者資訊列」上方，物品所屬分類旁邊已有 `MapPin`（lucide）圖示
  搭配縣市名稱，**建議在分類名稱前也加一顆對應分類色的小圖釘**（20px），跟卡片牆/首頁的圖釘
  是同一顆、同一色，讓使用者從列表點進詳情頁時視覺連續（這是定案預覽檔明講的「詳情頁沿用同一顆
  『傢俱大型物』棕色圖釘，讀者不用重新學一套符號」）。
- **抽籤結果（`LotterySection`）判斷**：抽籤狀態機（`open → drawing → awaiting_confirmation →
  confirmed/failed`，加上 `cancelled`）本質上跟交接進度一樣是**單向前進、有完整 audit log
  （`LotteryAuditLog`）佐證、且「重演驗證」正是這個功能的賣點之一**（deterministic shuffle
  可重演），非常符合戳章「留下不可逆紀錄」的敘事。**建議在 `LotterySection` 內部也用一條迷你
  StampTrack**（步驟：開放報名 → 已截止開獎 → 中選確認中 → 確定得主，`failed`/`cancelled`
  分支各自用 `stamp-void` 雙線邊框表示流標/取消，不是失敗的羞辱感而是「這次沒有成立、但過程
  留下了紀錄」的中性語氣）。這是**延伸戳章語言到一個定案預覽檔沒有示範、但本文件判斷邏輯上
  成立的地方**，理由已如上述；相對地，優惠券使用回報統計（`CouponUsageSection`）**不套用**
  戳章——那是「累積的社會證明統計」（幾則回報可用/失效），不是單一物件的線性進度，套用戳章
  會誤導成「這張券正在走某個流程」。
- **檢舉/申訴狀態**：物品詳情頁本身只有「檢舉這個物品」按鈕（`ReportButton`），**沒有顯示
  檢舉處理狀態的介面**（檢舉人要看自己檢舉的處理進度，目前給定的前台路由清單裡沒有對應頁面，
  只能透過通知得知結果）。本文件判斷：**這頁不套用戳章語言於檢舉/申訴**，因為根本沒有前台
  可視化這個流程的位置；若未來要新增「我的檢舉紀錄」頁面，屆時可以直接沿用戳章語言
  （檢舉狀態機 `submitted→triaged→in_progress→resolved/rejected→closed` 同樣是單向、有
  `AuditLog` 佐證，適用性判斷跟交接/抽籤一致），但這超出本次「28 個既定路由」的範圍，本文件
  只記錄這個延伸點供未來參考，不在本次規格內展開設計。

**降低操作摩擦**：

（多半是點擊觸發某個非破壞性操作，帳號刪除除外）。

**視覺套用**：不套用地圖/戳章語言（帳號設定是功能性頁面，非瀏覽/進度情境）。**唯一例外**：資料
匯出（`ExportDataSection`）與帳號刪除（`DeleteAccountSection`）**都是有明確狀態機的流程**
（匯出：`requested → processing → ready/failed`；刪除：`pending → cooling_off → executed`，
7 天冷卻期可撤銷）——這兩個狀態機是否適合戳章語言？**判斷：不套用**。理由：(1) 這兩個流程
發生頻率極低（一輩子可能就一兩次），使用者不需要「儀式感」的進度可視化，簡單的文字狀態
（「處理中」「已就緒，15 分鐘內有效」「冷卻期至 X 月 X 日，可撤銷」）加一個明確的倒數/期限
資訊更實用；(2) 帳號刪除的冷卻期本質是「可逆的等待」而非「不可逆的留痕前進」，跟戳章「蓋章
代表已留下紀錄」的敘事不符——刪除還沒真正發生前談「蓋章」語意上怪異。維持現有文字狀態 +
Badge 色塊即可。

**降低操作摩擦**：現有版面已經是分區塊（匯出/排行榜顯示/封鎖名單/刪除帳號各自一個 section）
的清楚結構，不需要調整。封鎖名單入口（M12 交付內容 3）目前是本頁裡的一個連結卡片而非獨立
`/me` 首頁卡片，**延續此決策**（CLAUDE.md 明確記載這是刻意的低頻功能收納位置）。

**降低學習摩擦**：「排行榜顯示」「封鎖名單」兩個區塊標題已經是白話文，不需調整。

**手機版轉換**：單欄分區塊版面天生適合手機。

**共用元件**：無新增。

action**：（本人）視情況留言；（moderator/admin）認領/轉換狀態/留言。

**視覺套用（本文件另一個需要明確取捨的判斷點）**：`open → in_progress → resolved/closed`
狀態機**本質上接近線性**（多數案例照順序走），但**允許跳步**（`open` 可以直接 `closed`，
不一定要經過 `resolved`）——這跟交接進度/抽籤（嚴格線性、每步都會發生）不同。**判斷：不套用
完整 StampTrack**，維持現有「頂端 Badge 顯示目前狀態 + 下方時間序事件清單（`events`，含
`fromStatus`/`toStatus`/留言/操作者）」的組合，理由：
1. 現有的 `events` 時間序清單本身**已經是一種戳章語言想達到的「留痕記錄」效果**（誰在什麼
   時候把狀態從 A 轉到 B、留了什麼話），只是用清單而非圖形化軌道呈現。**再疊加一條圖形化
   StampTrack 會造成同一份資料被畫兩次**（清單 + 軌道），不符合「不要為了套用而套用」的原則。
2. 若要視覺化，**更合理的做法是把現有 `events` 清單的視覺樣式往戳章語言靠**（每則事件前綴一個
   小型戳章圖示，用 `stamp-done`/`stamp-void` 兩態表示「這次轉換」還是「這次是留言、無狀態
   轉換」），而不是新增一條獨立軌道——這樣戳章造型變成「事件清單的裝飾強化」而非「新增一層
   重複資訊」。這是本文件建議的具體做法：**`events` 清單每一項前綴一顆小戳章**（`toStatus`
   存在時用 `stamp-done`，純留言則用中性小圓點，不套用完整戳章造型以免過重）。

**降低操作摩擦**：`AssignButton`（moderator/admin 認領）與 `TicketActions`（狀態轉換）**版面
位置維持不變**（已經是清楚的角色分工：本人看留言表單、moderator/admin 多看到認領/轉換操作）。

**降低學習摩擦**：`SUPPORT_TICKET_STATUS_LABEL`／`SUPPORT_TICKET_CATEGORY_LABEL` 已經是白話
中文對照，不需調整。

**手機版轉換**：既有單欄版面（標題/描述 → 附件縮圖 → 負責人 → 處理紀錄清單 → 操作表單）
天生適合手機，不需改動；附件縮圖網格（`flex-wrap`）在窄螢幕下維持現有 80×80px 尺寸即可
（多張截圖時允許換行）。

**共用元件**：StatusPill（1.2）、BackBar（1.5，已用）。
