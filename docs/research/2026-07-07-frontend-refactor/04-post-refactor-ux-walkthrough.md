## 環境限制說明（先講清楚，避免誤判成產品 bug）

1. **圖片視覺渲染**：本機用 `s3rver` 模擬 S3，`S3_PUBLIC_URL` 指向 `localhost:9200`；
   Next.js `next/image` 的內建 SSRF 防護會擋掉「解析到私有 IP」的上游圖片請求
   （`⨯ upstream image ... resolved to private ip ["127.0.0.1"]`），導致列表卡片與詳情頁
   的圖片顯示為破圖圖示。正式站 `S3_PUBLIC_URL` 是公開網域，不會有這個問題；上傳管線本身
   （含 HEIC 轉檔、thumb/medium 兩種尺寸產生）在伺服器端運作正常（`POST /api/uploads`
   均回 201），這純粹是本機走查的環境假影，不計入下面的問題清單。
2. **Next.js dev 模式浮動指示器**：`next dev` 在畫面左下角常駐一顆開發工具徽章
   （`nextjs-portal`），在 390px 窄螢幕會蓋住底部導覽最左側的「首頁」分頁，擋掉一般滑鼠
   點擊。這是 dev-only 元素，`next build`/`next start` 正式產物沒有這個東西，測試時已
   暫時關閉 `devIndicators` 驗證過點擊行為本身沒問題（見下方「導覽結構」），不計入問題清單。
3. 兩處計時偽陽性：感謝留言（walk-17/18）與優惠券揭露（walk-23）第一次截圖時 API 尚未
   回應完成（Turbopack 冷啟動編譯較慢，`e2e/tests/main-loop.spec.ts` 已有相同已知現象的
   註解），用更長等待重新驗證後兩者皆正確顯示，不是真的 bug。

## 逐流程結果

### 1. 未登入首訪（walk-01, walk-02）

✅ 順暢。首屏 3 秒內看得懂平台定位：「用不到的好物，剛好有人在找」+「台灣在地免費共享，
留言就有機會接手，不買賣、不交換」，下方三步驟圖解（上架分享／留言需要／私訊交接）進一步
說明。header 有明確「登入」CTA。未登入直接訪問 `/items` 可以正常瀏覽，沒有登入牆。目前無
物品時首頁顯示「目前還沒有物品上架，成為第一個分享好物的人吧」，文案得體。

### 2. 登入後首訪與初次導覽（walk-03, walk-04, walk-05）

✅ 順暢。登入後首次進站自動彈出 5 步導覽（`src/components/onboarding-tour.tsx`），每步
都是白話說明（「看到喜歡的，留言接手」「把用不到的分享出去」等），最後一步按鈕文案是
「開始使用」而不是生硬的「下一步／完成」。關閉後重新整理不再自動彈出（`localStorage`
`tour_done` 正常生效）。`/me` 頁「重新看一次導覽」按鈕可重新打開導覽，符合規格「可重開」
的要求。

### 3. 完整分享流程：實體物品（walk-06 ~ walk-18）

✅ 順暢，全程真的瀏覽器操作（含真的檔案上傳，非 API 捷徑）：

- 上架表單（`src/app/(shell)/items/new/item-form.tsx`）：標題／分享的話／分類／縣市／
  最多 5 張圖片，欄位標籤與 placeholder 清楚（例如「標題（2–60 字）」直接標明字數限制）。
  兩張圖片上傳完成（跳出 loading spinner）後送出按鈕才會從 disabled 轉為可點，防呆邏輯
  正確。
- 上架成功導向詳情頁，物品資訊、分享者暱稱正確顯示。
- 第二帳號在 `/items` 瀏覽到新物品、進入詳情頁留言（「留言表達你想要這個好物（第一則留言
  會自動被接受）」placeholder 清楚說明先到先得規則），送出後立即顯示「已被認領」，物主
  重新整理後看到「前往交接」按鈕。
- 私訊（`/conversations/[id]`）雙向訊息即時顯示，雙方各自標記完成後物品轉為「已完成
  分享」，接手者可留一則感謝留言（「感謝留言」欄位，限一次），物主重新整理後看得到。

### 4. 券類流程：優惠券（walk-19 ~ walk-27）

✅ 順暢。上架時選「優惠票券」分類會展開「優惠券資訊」子表單（面額／適用店家／到期日／
券碼，並註明「券碼會加密保存，只有接手者在交接開始後才看得到明文」）。認領＋交接開始後，
接手者在詳情頁看到「查看券碼」按鈕，點擊後即時打 `/api/items/[id]/coupon/reveal`
顯示明文券碼，經人工重試驗證解密內容與上架時輸入的一致（`GIFT-5566-TEST`）。完成交接後
「使用結果回報」區塊可回報可用／已失效，聚合統計即時更新（文案「僅供參考，平台不保證券碼
實際可兌換」符合 M9 規格的免責要求）。`/me/wallet` 正確分列「我分享的券」「我接手的券」，
接手者能在錢包看到剛接手的券。

### 5. 好康資訊 DealInfo（walk-28 ~ walk-32）

✅ 順暢。投稿表單（`src/app/(shell)/deal-infos/new/deal-info-form.tsx`）標題／自寫摘要／
來源連結／到期日／適用縣市（或全台適用）欄位清楚，摘要 placeholder 明確提醒「請用自己的話

見 `src/app/(shell)/deal-infos/[id]/page.tsx:63-66`）。

### 6. 導覽結構（walk-33 ~ walk-45）

✅ 順暢。底部導覽 5 格（首頁／逛好物／分享／訊息／我的）全部可點且導向正確路由；`/me`
中心頁 5 張卡片（優惠券錢包／訂閱通知／通知設定／帳號設定／我的個人頁）連結全部正確。
主題切換鈕（月亮/太陽圖示）點擊後 `<html>` 正確掛上 `dark` class，重新整理後維持深色

### 7. 空狀態與邊界（walk-46 ~ walk-51）

✅ 順暢。全新帳號（未做過任何操作）的訊息／通知／錢包／訂閱／問題回報五個列表頁，皆用
統一的 `EmptyState` 元件（圖示＋標題＋說明＋行動按鈕，見 `src/components/empty-state.tsx`）
呈現，文案具體且給出下一步引導（例如錢包空狀態「上架物品時選擇優惠券分類，就能把用不到的

再送出」（見 walk-51），錯誤訊息位置明顯、不需捲動就看得到，且表單內容（已上傳的圖片、

## 建議但未動的 P1 清單

1. **「食品雜貨」分類與「即期食品」規則耦合，介面沒有提示**（`src/lib/categories.ts`
   `EXPIRING_FOOD_CATEGORY_SLUG = "groceries"`；`prisma/seed.ts:36` 分類顯示名稱是
   「食品雜貨」）。使用者在 `/items/new` 選擇分類下拉選單裡看到的是「食品雜貨」，選下去
   卻自動展開一個標題為「即期食品規則」的子表單，強制填到期日並勾選「我確認這項食品完整
   包裝、未開封、常溫保存、尚未過期」——即使要分享的是離到期日還很久的食品（例如未拆封的
   調味料、罐頭），文案措辭仍是「即期食品」語境，容易讓人誤以為選錯分類。建議：分類顯示
   名稱與子表單標題二選一調整用詞一致（例如分類改名「食品雜貨（含即期食品）」，或子表單
   標題改成通用的「食品資訊」），這牽涉分類文案與既有測試斷言，不屬於「小修」範疇，需要
   一併確認下游影響再動。
2. **深色模式下的法定警示／優惠提示方塊對比與整體色調不協調**（`src/app/(shell)/items/
   new/item-form.tsx` 票券／點數子表單裡 `bg-brand-soft`＋`text-brand-ink` 的提示框，
   `legal-draft-notice.tsx` 同款寫法）。實測深色模式下（walk-52）這個提示框仍是淺色
   系統的高飽和青綠色底，在整體近黑背景中顯得突兀，雖然文字本身可讀，但視覺一致性差。
   `CLAUDE.md` 已記錄此為 M10 已知遺留，這次走查確認問題仍在；修法需要新增深色模式下的
   語意色 token（而非單點覆寫這幾處 class），建議與其餘品牌色 token 一併檢討，不在本次
   小修範圍內處理。
3. **多選 chip（好康資訊「適用縣市」等）觸控目標約 32px**，`CLAUDE.md` 已記載為刻意的
   WCAG 2.5.8（非 2.5.5）取捨，這次走查在 `/deal-infos/new` 實測確認現況與記錄一致，
   仍建議之後有餘裕時一併检討是否能在不破版的前提下加大。
